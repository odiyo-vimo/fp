1 What Is Boomi Managed File Transfer?

Boomi MFT is a cloud-native, governed file exchange platform built into the AtomSphere ecosystem. It provides a centralised hub through which files can be securely received from and delivered to external Trading Partners — businesses, government systems, or internal departments — without requiring each exchange to be hand-crafted in a Boomi Integration process.

MFT is not the same as using the SFTP V2 or FTP connectors inside a process. The fundamental difference is who owns the server:

Managed server endpoint✗ You must host your own✓ Boomi hosts the server for you
Audit & reporting dashboard✗ Build it yourself✓ Built-in MFT activity reports
Trading partner portal✗✓ Self-service partner onboarding

Trigger process on file arrivalVia Disk/SFTP listener (polling)Via Route trigger (event-driven)

2 Key Concepts
MFT Server

Boomi provisions a cloud-hosted file server on your behalf. No infrastructure to deploy or maintain. The server exposes endpoints for each enabled protocol (SFTP :22, FTPS :21, HTTPS :443) and handles TLS, key management, and authentication.

Trading Partners

A Trading Partner is a configured external entity that will send or receive files. Each has:

  • A unique username and credential (SSH key or password)
  • One or more assigned Source or Destination Routes
  • Optional IP whitelisting and access restrictions
Routes

A Route defines the path a file takes from arrival to destination. It maps an inbound drop folder to an action — either storing the file in an MFT-managed location or triggering a downstream Boomi Integration process.

MFT Connector (inside a Boomi Integration process)

An OOTB connector available inside Boomi Integration. Allows a running process to interact with the MFT platform — send a file to a Trading Partner's outbound folder, retrieve a file, list files waiting, or delete after processing.

4 Setting Up Boomi MFT — Step by Step
4.1 Enable MFT in Your AtomSphere Account
Licence required: Boomi MFT is a licensed add-on. If the MFT option is absent from the Services menu, contact your Boomi account representative to provision it.
1
Log in to Boomi AtomSphere at platform.boomi.com.
2
Navigate to Services → Managed File Transfer. The MFT dashboard appears here once enabled.
3
Contact your Boomi rep to enable MFT if not already provisioned. Once active, the full MFT menu is available.
4.2 Configure the MFT Server
1
Go to Managed File Transfer → Settings → Server Configuration.
2
Note your assigned MFT server hostname (e.g. mft-abc.boomi.com). Share this with Trading Partners as their connection endpoint.
3
Enable the required protocols: SFTP, FTPS, HTTPS, AS2. Enable only those your Trading Partners require.
4
Upload your organisation's TLS certificate for HTTPS/FTPS (or use the Boomi-provided certificate).
5
Export the SSH host key fingerprint to share with SFTP partners for their known_hosts file.
4.3 Create a Trading Partner
1
Go to Managed File Transfer → Trading Partners → New.
2
Enter a unique Partner Name (e.g. DWP-Supplier-A).
3
Choose authentication: Password (set a password) or SSH Public Key (paste the partner's public key). SSH key is strongly preferred.
4
Optionally whitelist the partner's IP addresses under Access Restrictions.
5
Save. The Trading Partner is now provisioned with login credentials.
4.4 Create a Source Route (Inbound)
1
Go to Managed File Transfer → Routes → New Source Route.
2
Select the Trading Partner created above.
3
Set the Source Folder — the virtual directory the partner uploads files into (e.g. /upload/dwp-a/).
4
Set the Action: either Store in MFT (file held for later retrieval) or Trigger a Boomi Process (immediate execution on arrival).
5
If triggering a process: select the deployed Boomi Integration process and map any Route Properties (filename, partner ID) to process properties.
6
Optionally set a File Filter (e.g. *.xml, *_PAYE_*.csv) so only matching files trigger the route.
7
Save and Deploy the route. The partner can now upload files to the configured folder.
4.5 Create a Destination Route (Outbound)
1
Go to Managed File Transfer → Routes → New Destination Route.
2
Select the Trading Partner.
3
Set the Destination Folder — the virtual directory the partner polls to collect files (e.g. /download/dwp-a/).
4
Set a File Retention Policy — how long files persist before automatic deletion (recommended: 3–7 days).
5
Save and Deploy. Boomi processes can now write files here using the MFT Connector Send action.
5 Using the MFT Connector in a Boomi Integration Process
5.1 Sending a File to a Trading Partner (Outbound)
Use this when your integration process has produced a file that a Trading Partner needs to collect.
1
Add a Connector step after your transform/mapping steps.
2
Connector Type: Boomi Managed File Transfer.
3
Create a Connection: set the MFT server URL and authenticate via API token or internal credentials.
4
Set the Action to: Send.
5
In the Operation: specify the Destination Route name and Trading Partner.
6
Set the File Name — static or dynamic via a Dynamic Document Property.
7
The current document in the process is written to the partner's outbound folder. The partner collects it via their configured protocol.
5.2 Retrieving a File from MFT (Get)
Use this when a process needs to pull a file uploaded by a partner and stored in MFT (route Action was "Store in MFT").
1
Start with a Schedule trigger or No Data start to run the retrieval process.
2
Add a Boomi MFT Connector step — Action: List — to check what files are available in the Source Route folder.
3
Use a Data Process or Decision shape to iterate over the list of file names.
4
For each file, use Action: Get to retrieve it. The file becomes the current document in the process.
5
Continue with mapping, routing, and delivery to the target system.
6
After successful processing, use Action: Delete to remove the file from the MFT server.
5.3 MFT Connector — Action Reference
ActionDescriptionWhen to Use
SendWrites the current document to a named Destination Route folderOutbound — making a file available for a partner to collect
GetRetrieves a specific file from a Source Route folder into the processInbound — pulling a stored file into a process
ListReturns a list of file names available in an MFT folderBefore Get — check what files are waiting
DeleteRemoves a file from an MFT folderPost-processing cleanup
MoveMoves a file within the MFT server (e.g. to an archive folder)Post-processing archival without deletion

7 Security Considerations
  • Use SSH public-key authentication for all SFTP Trading Partners in preference to passwords.
  • Enable IP whitelisting for partners whose source IPs are known and static.
  • Use FTPS (explicit TLS) or HTTPS for partners where SFTP is not supported — never plain FTP in production.
  • Rotate MFT server TLS certificates on your standard certificate renewal cycle.
  • Set short File Retention Policies (3–7 days) on Destination Routes to avoid sensitive data lingering on the MFT server.
  • Review Transfer Logs weekly and configure Notifications for failed transfers to catch connectivity issues early.
  • For AS2, ensure MDN (Message Disposition Notification) acknowledgements are configured for all routes to provide non-repudiation.
  • Restrict MFT admin access using Boomi AtomSphere role-based access control — not all Integration developers need MFT admin rights.
8 Recommended Integration Patterns
PatternScenarioImplementation
Trigger-on-ArrivalPartner uploads file; process must start immediatelySource Route Action = Trigger Boomi Process. Process receives the file as its start document.
Scheduled PollPartner uploads files throughout the day; process runs at fixed intervalsSource Route Action = Store in MFT. Scheduled process: MFT Connector List → Get → process → Delete.
Outbound BatchProcess generates a report/extract; partner collects at their own scheduleProcess uses MFT Connector Send to write to Destination Route. Partner polls via SFTP/HTTPS on their own schedule.
Error & RetryFile processing fails; file must not be lostSource Route stores file. On process failure, file remains in MFT until explicitly deleted. Retry by re-triggering the Get step.
Multi-Partner Fan-outSame file must be sent to multiple partners simultaneouslyProcess branches to multiple MFT Connector Send steps, each targeting a different Destination Route / Trading Partner.
9 Troubleshooting Common Issues

10 MFT vs SFTP Connector — Quick Reference
Rule of thumb: Use Boomi MFT when you need a managed, auditable file exchange server for external Trading Partners. Use the SFTP V2 Connector when you need to connect to a third-party SFTP server that you do not control.

✅ Use Boomi MFT when…

  • You need to provide an SFTP/FTPS/HTTPS endpoint to an external partner
  • You require a built-in audit trail and transfer dashboard
  • You want event-driven trigger-on-arrival without polling
  • You need partner-level authentication and access control without building it yourself
  • You need AS2 with MDN acknowledgements
  • You need a file retention / archival policy on the exchange point

✅ Use SFTP V2 Connector (in a process) when…

  • You are connecting to a third-party SFTP server you do not own
  • You need fine-grained control over the SFTP session (key types, ciphers, known_hosts)
  • You are integrating with a legacy government SFTP endpoint
  • MFT is not part of your Boomi licence
  • The exchange is internal (same estate) and governance overhead is not needed
