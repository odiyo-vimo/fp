How to use this page

This page is the single reference for the eight-week plan to onboard the Boomi Managed File Transfer (MFT) programme and deliver the first MFT interface into the Development environment — across four two-week sprints, from Discovery & Mobilisation through to a working, deployed interface.

  • Read top-to-bottom for the narrative: at-a-glance summary → live roadmap → sprint-by-sprint detail → user stories → backlog → ways of working → RAID.
  • Download the artefacts from the sticky bar at the very top: the Word delivery plan, the Excel sprint backlog (all stories with acceptance criteria and points), the draw.io roadmap, or everything as a single zip.
  • Two workstreams run in parallel throughout — MFT Interface Delivery (design, build, deploy the pilot interface) and Intelligrator (the self-service provisioning portal). User stories are tagged by workstream.
  • Design principle: provisioning is configuration-driven — one generic MFT engine process reads a config table with one row per interface. New interfaces are added as config rows, not by generating per-request Boomi components.
Context. Boomi's native MFT is based on the Thru, Inc. technology acquired in May 2025: the postal-service model of Organizations, Flows and Endpoints, with Automated File Transfer, Management and Data channels, and monitoring via the Activity Page. This plan assumes one pilot interface in scope for the eight weeks.

Sprint 1 · Weeks 1–2 — Discovery & Mobilisation

Sprint goal. Stand up the squad and access, understand the current-state file-transfer landscape, agree the pilot interface, and establish the target MFT operating model on Boomi. Confirm Intelligrator MVP scope and personas.

Key activities

  • Team onboarding, RACI, ways of working and ceremony cadence agreed
  • Environment access requested: Boomi platform, Dev runtime (Atom/Molecule) and native MFT capability enablement
  • Current-state discovery: inventory of existing file interfaces (endpoints, protocols, schedules, volumes, file formats, encryption)
  • Pilot interface scored, shortlisted and selected
  • Target MFT operating model documented — Thru-based postal model (Organizations / Flows / Endpoints), Automated File Transfer vs File Sharing, Management vs Data channels, naming and Activity-Page monitoring conventions
  • Intelligrator vision, MVP boundary and personas (Requester, MFT Admin/Approver, Platform Engineer, Auditor) confirmed

Deliverables

  • Current-state file-transfer inventory
  • Pilot interface selection & scoring
  • Mobilisation pack & RACI
  • MFT operating model v0.1
  • Intelligrator MVP scope statement
Milestone: M1 — Discovery complete, pilot selected, Dev + MFT access secured.

Sprint 2 · Weeks 3–4 — Solution Design & Environment Readiness

Sprint goal. Produce the detailed design for the pilot MFT interface, provision and secure the Dev environment and connectivity, and design the Intelligrator config-driven provisioning model, wizard, approval queue and lifecycle.

Key activities

  • Pilot interface solution design (SDD): Organizations, Flow, source/target Endpoints, protocol (SFTP/AS2), schedule, transformation (if any), error handling
  • Connectivity & network readiness: firewall rules, routes, SFTP host keys, IP allow-listing in Dev
  • Security & key-management design: PGP handling, credential vaulting, data-classification controls in transit and at rest
  • Config-driven engine design: single generic MFT engine process driven by a config table (one row per interface) — preferred over per-request component generation
  • Monitoring & alerting design against the Activity Page, with alert thresholds and SLAs
  • Intelligrator: data model, provisioning config schema, wizard field spec, approval queue design and lifecycle state machine

Deliverables

  • Pilot interface Solution Design Document (SDD)
  • Environment & connectivity readiness checklist
  • Config/cross-reference schema & data model
  • Intelligrator design pack (wizard + approval + states)
Milestone: M2 — Solution design signed off; environment & connectivity ready.

Sprint 3 · Weeks 5–6 — Build & Configure

Sprint goal. Build and configure the pilot MFT interface using the config-driven engine, and build the Intelligrator MVP (wizard → approval → provisioning trigger).

Key activities

  • Configure Boomi MFT Organizations, Flow and source/target Endpoints for the pilot
  • Build the generic MFT engine process, parameterised from the config table
  • Populate the config/cross-reference row for the pilot interface
  • Implement error handling, retries, dead-letter handling, notifications and logging
  • Wire up monitoring & alerting via the Activity Page in Dev
  • Intelligrator MVP: wizard workflow, approval queue, config-write on approval and provisioning trigger into the generic engine
  • Unit-test the configured components

Deliverables

  • Configured, unit-tested pilot MFT interface
  • Generic MFT engine process
  • Intelligrator MVP build (wizard + approval + provisioning trigger)
  • Unit-test evidence
Milestone: M3 — Pilot built & unit-tested; Intelligrator MVP built.

Sprint 4 · Weeks 7–8 — Test, Deploy to Dev & Handover

Sprint goal. End-to-end SIT of the pilot interface, deploy to Development, produce the runbook, and demonstrate Intelligrator provisioning into Dev end-to-end. Decide Dev go/no-go and shape the Phase-2 backlog.

Key activities

  • End-to-end SIT: automated file transfer source → Boomi MFT → target; positive, negative, encryption, large-file and resend cases
  • Non-functional checks: throughput, file size, schedule adherence
  • Deploy the pilot interface to Development (config row / packaged component)
  • Operational runbook & support model (start/stop, reprocessing, alerts, escalation)
  • Intelligrator: lifecycle transitions (Requested → Approved → Provisioned → Active → Retired), audit log, and provisioning-into-Dev demonstrated
  • Sprint review & Dev go/no-go; retrospective; prioritised Phase-2 backlog

Deliverables

  • SIT results & defect log
  • Pilot MFT interface live in Development
  • Operational runbook
  • Intelligrator end-to-end provisioning demo + audit trail
  • Phase-2 backlog & recommendations
Milestone: M4 — MFT interface LIVE in Dev; auto-provisioning demonstrated.

7. User stories

Stories are tagged by workstream — MFT Interface Intelligrator Programme — with a MoSCoW priority and an indicative point estimate. Full detail is also in the downloadable Excel backlog.

Sprint 1 — Discovery & Mobilisation

S1-01 Programme Must 3 pts · Delivery Lead

Team onboarding & ways of working

As a delivery lead, I want the squad onboarded with a RACI and agreed ceremonies so that the team can deliver predictably.

Acceptance criteria:
  • Squad, roles and RACI published
  • Ceremony cadence and Definition of Done agreed
  • Backlog tool and boards set up
S1-02 MFT Interface Must 5 pts · Platform Engineer

Boomi Dev environment & MFT access

As a Platform Engineer, I want access to the Boomi platform with the Dev runtime and native MFT capability enabled so that I can begin configuration.

Acceptance criteria:
  • Boomi accounts provisioned with correct roles
  • Dev Atom/Molecule runtime available
  • MFT capability enabled and reachable
S1-03 MFT Interface Must 8 pts · Integration Architect

Current-state file-transfer discovery

As an Integration Architect, I want an inventory of existing file-transfer interfaces (endpoints, protocols, schedules, volumes, encryption) so that I can size and prioritise migration.

Acceptance criteria:
  • Inventory captures source/target, protocol, schedule, volume, format, encryption
  • Interfaces classified by complexity and risk
  • Inventory reviewed with stakeholders
S1-04 MFT Interface Must 3 pts · Product Owner

Pilot interface selection

As a Product Owner, I want a scored shortlist and a chosen pilot interface so that Sprint 2 design has a clear target.

Acceptance criteria:
  • Scoring criteria agreed (risk, value, complexity)
  • Shortlist scored
  • Pilot selected and signed off
S1-05 MFT Interface Should 5 pts · Integration Architect

MFT operating model & guardrails

As an Integration Architect, I want the target MFT operating model documented (postal model roles, AFT vs File Sharing usage, channels, naming) so that all interfaces follow consistent standards.

Acceptance criteria:
  • Postal model (Organizations/Flows/Endpoints) usage defined
  • AFT vs File Sharing selection criteria documented
  • Naming, channels and monitoring conventions agreed
S1-06 Intelligrator Must 3 pts · Product Owner

Intelligrator vision & MVP scope

As a Product Owner, I want the Intelligrator MVP scope and personas confirmed so that build stays bounded.

Acceptance criteria:
  • MVP boundary agreed (in/out of scope)
  • Personas confirmed
  • Success measures defined
S1-07 Intelligrator Should 3 pts · MFT Admin

Persona & permission model

As an MFT Admin, I want personas (Requester, Approver, Platform Engineer, Auditor) and their permissions defined so that access is least-privilege.

Acceptance criteria:
  • Personas mapped to permissions
  • Least-privilege model documented
  • Segregation of duties preserved (requester ≠ approver)

Sprint 2 — Solution Design & Environment Readiness

S2-01 MFT Interface Must 8 pts · Integration Architect

Pilot interface solution design

As an Integration Architect, I want a detailed SDD for the pilot (Organizations/Flow/Endpoints, protocol, schedule, transformation, error handling) so that build is unambiguous.

Acceptance criteria:
  • SDD covers full transfer path and endpoints
  • Protocol, schedule and error handling specified
  • Design reviewed and signed off
S2-02 MFT Interface Must 5 pts · Platform Engineer

Connectivity & network readiness

As a Platform Engineer, I want firewall rules, routes and SFTP host keys provisioned in Dev so that endpoints are reachable.

Acceptance criteria:
  • Firewall/route requests raised and confirmed
  • SFTP host keys exchanged
  • Connectivity verified from Dev runtime
S2-03 MFT Interface Must 5 pts · Security Architect

Security & key-management design

As a Security Architect, I want PGP/credential handling and data-classification controls designed so that files are protected in transit and at rest.

Acceptance criteria:
  • PGP key lifecycle defined
  • Credential vaulting approach agreed
  • Data-classification handling documented
S2-04 MFT Interface Must 8 pts · Integration Architect

Config-driven engine design

As an Integration Architect, I want a single generic MFT engine process driven by a config table (one row per interface) so that new interfaces are provisioned without per-request component builds.

Acceptance criteria:
  • Config schema defines one row per interface
  • Generic engine behaviour specified
  • No per-request component generation required
S2-05 MFT Interface Should 5 pts · Operations Analyst

Monitoring & alerting design

As an Operations Analyst, I want monitoring via the Activity Page plus alerts and SLA thresholds designed so that failures are detected quickly.

Acceptance criteria:
  • Monitoring approach mapped to Activity Page
  • Alert thresholds and channels defined
  • SLA targets agreed
S2-06 Intelligrator Must 5 pts · Platform Engineer

Intelligrator data model & config schema

As a Platform Engineer, I want the Intelligrator data model and provisioning config schema defined so that requests map cleanly to engine config rows.

Acceptance criteria:
  • Data model covers request, approval and interface entities
  • Request fields map to engine config columns
  • Schema reviewed
S2-07 Intelligrator Must 5 pts · Requester

Wizard field specification

As a Requester, I want a guided wizard capturing all details needed to provision a transfer (source, target, protocol, schedule, format, encryption) so that I can self-serve a request.

Acceptance criteria:
  • All mandatory provisioning fields captured
  • Validation rules defined
  • Wizard steps and help text specified
S2-08 Intelligrator Must 5 pts · Approver

Approval queue & lifecycle design

As an Approver, I want an approval queue and a defined lifecycle (Requested → Approved → Provisioned → Active → Retired) so that governance is enforced.

Acceptance criteria:
  • State machine documented with allowed transitions
  • Approval queue behaviour defined
  • Rejection and rework paths covered

Sprint 3 — Build & Configure

S3-01 MFT Interface Must 8 pts · Platform Engineer

Configure pilot Organizations/Flow/Endpoints

As a Platform Engineer, I want the pilot's Boomi MFT Organizations, Flow and source/target Endpoints configured so that a transfer path exists.

Acceptance criteria:
  • Organizations and Flow created
  • Source and target Endpoints configured
  • Path validated in Dev
S3-02 MFT Interface Must 8 pts · Platform Engineer

Build generic MFT engine process

As a Platform Engineer, I want the parameterised engine process built and reading from the config table so that interfaces run from configuration.

Acceptance criteria:
  • Engine reads interface config at runtime
  • Pilot config row drives the transfer
  • No hard-coded, per-request logic
S3-03 MFT Interface Must 5 pts · Operations Analyst

Error handling, retries & notifications

As an Operations Analyst, I want retries, dead-letter handling and failure notifications implemented so that transient errors don't cause data loss.

Acceptance criteria:
  • Retry policy implemented
  • Failed files routed to dead-letter handling
  • Notifications raised on failure
S3-04 MFT Interface Should 3 pts · Operations Analyst

Monitoring & alerting build

As an Operations Analyst, I want the Activity Page monitoring and alerts wired up so that I can see transfer status in Dev.

Acceptance criteria:
  • Transfers visible on the Activity Page
  • Alerts fire on failure/SLA breach
  • Status verified during unit test
S3-05 MFT Interface Must 3 pts · Platform Engineer

Unit-test the pilot interface

As a Platform Engineer, I want the configured interface unit-tested so that components behave as designed before SIT.

Acceptance criteria:
  • Test cases cover happy and error paths
  • Results recorded
  • Defects logged and triaged
S3-06 Intelligrator Must 8 pts · Requester

Wizard workflow build (MVP)

As a Requester, I want to complete and submit an MFT request through the wizard so that it enters the approval queue.

Acceptance criteria:
  • Wizard captures and validates all fields
  • Submission creates a request in Requested state
  • Request appears in the approval queue
S3-07 Intelligrator Must 5 pts · Approver

Approval queue build (MVP)

As an Approver, I want to review, approve or reject requests from a queue so that only governed transfers proceed.

Acceptance criteria:
  • Queue lists pending requests with detail
  • Approve/reject with comments supported
  • State updates on decision
S3-08 Intelligrator Must 8 pts · Platform Engineer

Config-write provisioning trigger

As a Platform Engineer, I want approved requests to write a config row and trigger the generic engine so that provisioning is automated (no per-request components).

Acceptance criteria:
  • Approval writes a config row
  • Generic engine picks up the new interface
  • No per-request component is generated

Sprint 4 — Test, Deploy to Dev & Handover

S4-01 MFT Interface Must 8 pts · Test Analyst

End-to-end SIT

As a Test Analyst, I want an end-to-end automated file transfer tested (positive, negative, encryption, large file, resend) so that the interface is proven.

Acceptance criteria:
  • Positive and negative cases pass
  • Encryption and large-file cases pass
  • Resend/reprocess verified; results recorded
S4-02 MFT Interface Must 5 pts · Platform Engineer

Deploy interface to Development

As a Platform Engineer, I want the pilot interface deployed and running in Dev so that it is operational for integrated testing.

Acceptance criteria:
  • Interface deployed to Dev
  • Scheduled transfer runs successfully
  • Deployment recorded/repeatable
S4-03 MFT Interface Should 3 pts · Operations Analyst

Operational runbook & support model

As an Operations Analyst, I want a runbook (start/stop, reprocessing, alerts, escalation) so that the interface can be supported.

Acceptance criteria:
  • Runbook covers operate, monitor and recover
  • Escalation path defined
  • Reviewed with support
S4-04 Intelligrator Should 5 pts · Requester

Lifecycle transitions & status visibility

As a Requester, I want to see my request move through its lifecycle states so that I know when the transfer is live.

Acceptance criteria:
  • State shown to requester
  • Transitions reflect real provisioning status
  • Active state confirmed on go-live
S4-05 Intelligrator Must 3 pts · Auditor

Audit log

As an Auditor, I want an immutable audit trail of who requested, approved and provisioned each transfer so that governance is evidenced.

Acceptance criteria:
  • Request, approval and provisioning events logged
  • Actor and timestamp captured
  • Log is tamper-evident/immutable
S4-06 Intelligrator Must 5 pts · Product Owner

Provision-into-Dev demo (end-to-end)

As a Product Owner, I want an end-to-end demo (request → approve → auto-provision → active in Dev) so that the MVP value is proven.

Acceptance criteria:
  • Full flow demonstrated end-to-end
  • Interface active in Dev after approval
  • Stakeholder sign-off captured
S4-07 Programme Must 3 pts · Delivery Lead

Go/No-go & Phase-2 backlog

As a delivery lead, I want a Dev sign-off decision and a prioritised Phase-2 backlog so that the next phase is ready to start.

Acceptance criteria:
  • Go/no-go decision recorded
  • Retro actions captured
  • Phase-2 backlog prioritised

RetrospectiveLast dayDelivery teamInspect and adapt ways of working.

Definition of Ready

  • Story has a clear user-value statement and acceptance criteria
  • Dependencies and test data identified
  • Sized by the team and small enough for one sprint
  • No unresolved blocking questions

Definition of Done

  • Acceptance criteria met and demonstrated
  • Built config-driven (no per-request components) where applicable
  • Unit-tested; defects logged/triaged
  • Peer-reviewed and documented
  • Deployed to the target environment for the sprint
  • Published web page uses generic programme terms only (governance naming rule)
