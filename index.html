2Executive summary

Boomi acquired Thru, Inc. in May 2025 to embed enterprise-grade Managed File Transfer natively within the Boomi Enterprise Platform, unifying API-, application- and file-based data movement under a single control plane.

This SDD sets out how Boomi native MFT delivers governed, secure, auditable and config-driven file transfer for the estate, coexisting with and progressively replacing legacy WebMethods and bespoke file-mover jobs. The design is anchored to the programme's inviolable principle — “config over construction”: a single generic MFT engine process handles every transfer, and each approved interface becomes a configuration row in a routing table, never a new Boomi component. This keeps the estate small, uniform, testable and cheap to govern as onboarding scales through the Intelligrator self-service portal.

Recommendation to the Design Authority

Adopt Boomi native MFT (Thru) as the strategic MFT capability, deployed as a hybrid topology — Boomi-hosted control plane plus customer-managed MFT runtimes so OFFICIAL payloads remain within the trusted boundary.

Proceed on the basis that the design decisions (§12) are accepted and the seven open questions (§14) are assigned named owners before build — most materially: data residency/assurance for OFFICIAL, long-term audit retention beyond native defaults, and native protocol coverage versus partner requirements.

To provide the design baseline for MFT on Boomi sufficient for Design Authority sign-off: the capabilities relied upon, the target architecture, how it integrates with the estate, the non-functional posture, and an honest assessment of gaps and the actions they drive.

4Background & context

The estate has historically moved files through a mix of WebMethods Integration Server flows, ChRIS, and bespoke scheduled SFTP scripts — point-to-point between systems of record such as ETMP, NPS and DES, and outward to other government departments, agents and financial institutions. This fragmentation produces inconsistent security controls, weak central audit, and a per-interface bespoke build cost that does not scale.

Drivers

  • Consolidate file movement onto Boomi iPaaS as the strategic platform.
  • Replace bespoke code with configuration, cutting build and assurance cost.
  • Provide a single source of truth for monitoring, tracking and audit.
  • Enable governed self-service onboarding via Intelligrator.
  • Provide a controlled path to decommission WebMethods.

6Boomi MFT capabilities (Thru)

Boomi MFT is a storage-based, cloud-native service (from the Thru acquisition) providing persistence, scheduling, fault tolerance, monitoring, audit logging and security, integrated directly with Boomi Integration processes so file flows and processing flows are managed together.

  • Protocols & endpoints: SFTP, FTPS and AS2 under Organizations → Endpoints → Flows; MFT SFTP server on port 22 with full SSH authentication.
  • Trading-partner connectivity: partner/community model; Boomi is extending its unified trading-partner framework to MFT.
  • Persistence & guaranteed delivery: files persisted with automatic retry and fault tolerance.
  • Event-driven integration: file arrival triggers a Boomi process; the MFT Connector bridges file and processing flows.
  • Security: PGP encrypt/decrypt incl. ASCII-armor (.asc); TLS/SSH in transit; platform RBAC.
  • In-flight handling: TAR archive/extract (optional compression) in Flow Endpoints; filename-pattern matching and post-actions.
  • Tracking & audit: end-to-end visibility with delivery confirmation — a single source of truth for all transfers.
  • Scheduling & automation: time-based automated transfer with schedule history.
  • Observability: OpenTelemetry export of metrics, logs and traces; runtime heartbeat/health telemetry.
  • Hybrid runtime: MFT cloud components + on-prem MFT runtimes + MFT Connector inside processes.

Recent platform movement — validate against the live release

The March 2026 release added TAR archive support, PGP ASCII-armor, continued (deprecated) SFTP DSA host-key support for legacy partners, automated data retention (schedule history 3 months, audit logs 12 months), and runtime performance/health-telemetry improvements.

The Q2 2026 roadmap focuses on scaling the distributed hybrid MFT architecture and extending the trading-partner framework and file-flow extensibility — confirm the exact GA feature set at build time.

7.2 Config over construction — the generic engine

A single generic MFT engine process is the only file-moving component. Each approved interface is a configuration row: source/target endpoints, protocol, PGP key references, schedule, filename pattern, post-actions. Onboarding adds a row; it never adds a component.

Design guardrail

Any requirement that cannot be expressed as configuration (e.g. bespoke content-based routing or complex per-file transformation) must be raised as a deviation to the Design Authority, with a named owner, before it is built — it is the point at which “construction” risks creeping back in.

Inbound (partner → estate)

  • Partner pushes to / engine pulls from an MFT endpoint; partner authenticated by SSH key / certificate.
  • File lands in a transient encrypted landing zone; a file-arrival event triggers the engine.
  • Engine reads the config row, decrypts (PGP), validates, de-archives, routes/transforms, delivers via connector.
  • Delivery confirmed; transfer, receipt and audit written to the tracking log.

Outbound (estate → partner)

  • Source drops a file / engine polls on schedule; engine picks up the config row.
  • Engine encrypts (PGP/ASCII-armor), optionally archives (TAR), delivers to partner endpoint.
  • Delivery receipt captured, audit written; failures retried and alerted.

7.4 Key & certificate management

SSH keys, PGP keypairs and TLS certificates are held in the platform key/certificate store and referenced by configuration. Custody, rotation and enterprise HSM/KMS integration are a security dependency (§14).

8Integration with the estate

8.1 Connectivity to systems of record

The generic engine reaches ETMP, NPS, DES, ChRIS and HoD services through Boomi connectors (SFTP, database, JMS/queue, HTTP) from the trusted runtime zone. Payloads stay within the boundary; only control/tracking metadata surfaces to the control plane.

8.2 WebMethods coexistence & decommission

  • Boomi MFT can consume from and deliver to WebMethods drop-zones, migrating interfaces one config row at a time.
  • Each migrated interface is proven in parallel, then WebMethods is retired for that flow — controlled, low-risk, no big-bang.
  • No automated WebMethods→Boomi converter exists; migration is deliberate re-configuration (gap G-12).

8.3 Intelligrator (self-service onboarding)

Intelligrator is the governed front door: an approved interface request writes a new configuration row to the routing table consumed by the generic engine. It enforces the approval gate and standards, and creates no Boomi components — preserving config-over-construction end to end.

8.4 Identity, monitoring & key custody

  • Operator access via enterprise IdP/SSO (later phase); partner authentication by SSH key / certificate.
  • Operational telemetry exported via OpenTelemetry to the enterprise SIEM (e.g. Splunk).
  • Key custody integrates with the enterprise HSM/KMS (to be confirmed).

10Non-functional design

Availability & resilience

Clustered Boomi Molecule runtime for HA and fault tolerance; load-balanced edge. Gateway HA topology to confirm (§14).

Disaster recovery

Secondary runtime with replicated configuration. RTO/RPO to be set; the config/routing table and key material are the critical state to protect.

Scalability & performance

Cloud components scale elastically; Atoms scale horizontally. March 2026 runtime optimisations reduce retry cycles/latency. Benchmark vs WebMethods (G-05).

Security & data protection

In transit TLS/SSH; at rest PGP; transient zones encrypted. RBAC and segregation of duties. Key custody via HSM/KMS (confirm). Hybrid runtime keeps payload in boundary.

Auditability & retention

Immutable tracking/audit record per transfer. Native retention defaults (audit 12 mo, schedule history 3 mo) are likely below statutory needs — export to long-term archive (G-02b).

Observability & operations

OpenTelemetry export to SIEM; heartbeat/health telemetry for proactive monitoring and SLA-breach alerting.

13Risks, assumptions, issues, dependencies

Risks

  • Product maturity/roadmap dependence (G-01) could delay relied-upon features.
  • Data residency/assurance for OFFICIAL unresolved could block go-live (G-02a).
  • Throughput at estate batch volumes unproven (G-05).

Issues

  • Native audit retention default (12 months) below likely statutory requirements — open until archive design agreed.

Assumptions

  • A UK-region/assured control-plane option exists for OFFICIAL.
  • Partners predominantly support SFTP/FTPS/AS2.
  • Enterprise SIEM can ingest OTel; HSM/KMS integration is feasible.

Dependencies

  • Network/firewall low-level design (DMZ ↔ trusted zone).
  • Enterprise IdP/SSO for operator access (later phase).
  • Enterprise HSM/KMS for key custody.
  • Intelligrator for governed config authoring.
