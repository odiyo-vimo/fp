1 Intent

P-116 addresses high-volume, scheduled or managed bulk file exchange where per-record API calls would be prohibitively expensive, wasteful, or simply unavailable. Typical use cases include overnight reconciliation files, statutory reporting extracts (RTI, CIS, SA), bulk uploads from third-party providers, and file transfers from taxpayers or downstream consumers.

The pattern differs from a plain SFTP connector call in that it adds operational concerns: integrity verification via checksum, row-count validation, PGP encryption/decryption, retry, archiving, and a full audit trail — the features that distinguish managed transfer from raw file copy.

Key principle: The iPaaS is the broker, not the endpoint. MFT places the iPaaS between sender and receiver, enforcing validation and governance without either party needing to be available simultaneously.
2 Context & Problem

Large datasets cannot be passed record-by-record through synchronous APIs without overwhelming rate limits, incurring excessive latency, or exceeding payload caps. Legacy government systems only accept bulk files over SFTP or MFT protocols. Third-party providers produce CSV/XML extracts that must be ingested on a schedule.

Without a managed pattern, ad hoc SFTP scripts typically have no:

  • Integrity checking — truncated or corrupted files go undetected
  • Retry logic — transient network failure causes silent data loss
  • Audit trail — no record of what was transferred, when, or to whom
  • Encryption lifecycle — PGP keys expire, files sent in clear text
  • Alerting — nobody knows a file was missed until the next business day
3 Solution

The P-116 pattern structures file transfer as a governed pipeline with five logical stages:

  1. Acquire — collect the file from source (SFTP, FTPS, Boomi MFT, Disk, AS2)
  2. Validate — verify checksum (MD5/SHA-256), row count, file format, and mandatory fields
  3. Transform — decrypt (PGP), parse (CSV/XML/fixed-width), map to canonical or target schema
  4. Deliver — route to one or more targets (DB, REST API, downstream SFTP, Boomi MFT outbound)
  5. Archive & Audit — move source file to archive folder, write audit DB record, send confirmation

6 File Integrity & Validation
6.1 Checksum Validation

A companion checksum file (.md5, .sha256) or manifest must be provided by the source. The Boomi process must read the companion file first, compute the hash of the received file via Groovy (MessageDigest API or Bouncy Castle), compare expected vs actual, and log both values in the audit record.

If the source does not provide a checksum: implement a minimum row-count check using a trailer record or line count. Never deliver a file to a downstream system without some integrity verification.
6.2 Row Count Validation

Many bulk file formats include a header or trailer record with a total row count. Extract the expected count, split and count data records using the Data Process shape, and compare before delivery. A partial file delivery is worse than no delivery.

6.3 Schema / Format Validation

Where an XSD or JSON Schema is available, validate before transformation. For CSV, validate mandatory column presence and data type conformance in the Groovy script stage. Reject malformed records to an error output rather than failing the whole batch where possible.

8 Error Handling & Retry
8.1 Dead-Letter Folder

Any file that fails validation or delivery must be written to /deadletter/YYYY-MM-DD/, recorded in the audit table with status = FAILED, and subject to an immediate ops alert.

8.2 Retry Strategy (delivery failures only)
  • Attempt 1: immediate
  • Attempt 2: +2 minutes
  • Attempt 3: +8 minutes
  • Attempt 4: Dead-letter + alert

Implement via Try/Catch wrapping the delivery connector with a counter in a Dynamic Process Property. Do not retry validation failures — a corrupted file will not become uncorrupted on retry.

8.3 Partial Failure (Record-Level Errors)

Where the file is split into individual records, process each independently. Route errored records to a parallel error output so valid records are still delivered. Raise an alert with the count of failed records.

Rule: Never silently swallow errors. Every failure must produce a dead-letter write, an audit record update, and an alert — ideally all three.

10 When to Use / When to Avoid

✅ When to Use

  • High-volume bulk data (thousands to millions of records) where per-record APIs are impractical
  • Source or target only supports file-based protocols (SFTP, FTPS, AS2, Disk)
  • Overnight reconciliation, statutory reporting extracts (RTI, CIS, SA), payroll files
  • External Trading Partner exchange needing a managed, auditable endpoint
  • Files require integrity verification (checksum) before downstream delivery
  • PGP encryption/decryption is required as part of the exchange
  • Sender and receiver availability must be decoupled (async handoff)

❌ When to Avoid

  • Real-time, sub-second requirements — use P-101 (REST Inbound) or P-121 (Event Subscribe)
  • Small payloads (<100 records) where a direct REST API call is available and appropriate
  • Source and target are both Boomi-native — use Message Queue or Event Streams instead
  • Data must be validated record-by-record with immediate per-record feedback to the caller
  • File formats are unknown / unstructured and cannot be parsed by the Boomi Data Process shape
11 Boomi Platform Features Used
Boomi MFT Connector

Managed server endpoint for Trading Partner exchange. Configure server hostname, TLS cert, SSH key; create Trading Partners and Source/Destination Routes.

SFTP V2 Connector

Connect to third-party SFTP servers as source or archive destination. SSH public-key auth; private key in Boomi Certificate Management; known_hosts enforced.

Data Process Shape (Split)

Divide multi-record file into individual documents for record-level processing. Split by line count (CSV) or XML element. Use with Try/Catch for per-record error isolation.

Custom Scripting (Groovy)

PGP decrypt/encrypt and checksum computation. Uses Bouncy Castle API bundled in Boomi. Passphrase via getEncryptedProperty().

Map Shape

Transform parsed records to canonical or target schema. Embed lookup calls to database via function in the map for reference data enrichment.

Try/Catch Shape

Wrap delivery connector; capture exceptions; route to dead-letter branch. Pass error message to Dynamic Process Property for audit INSERT.

Dynamic Document Properties

Carry file metadata (filename, source, checksum, row count) through all process steps. Set on acquisition; read in the audit INSERT step.

Encrypted Process Properties

Store PGP passphrase, SFTP passwords, API keys. Never use plain-text process properties for secrets.

Database Connector (JDBC)

Audit INSERT; configuration lookup; target delivery via batch INSERT. Use UPSERT for idempotent re-runs.

Schedule Shape

Cron-based process trigger. Schedule multiple windows. Disable during maintenance via environment extension.

Route Shape

Branch delivery to multiple targets based on file type, source partner, content fields, or Dynamic Document Property values.

Process Reporting

Execution history and error investigation. Cross-reference with audit table using boomi_process_id for end-to-end traceability.
