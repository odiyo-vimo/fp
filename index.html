How to use this page

This page is the single reference for the Intelligrator product — a self-service portal where users request scheduled Managed File Transfer (MFT) interfaces, an admin approves them from a queue, and Boomi automatically builds and deploys the interface to the Dev environment.

  • Download bar (top): grab the Word product specification (for circulation and review), the draw.io architecture diagram (editable in app.diagrams.net via File → Open From → Device), or the full zip bundle.
  • Left navigation: jumps to each section; the active section is highlighted as you scroll.
  • User stories (§5): written in standard As-a / I-want / So-that format with Given-When-Then acceptance criteria — ready to paste into Jira/ADO. Priorities use MoSCoW badges.
  • Design authority sections (§6–§8): the recommended build approach (template + configuration-driven) and the patterns that justify it. This aligns with the P-116 MFT pattern in the wider integration catalogue.

1. Product overview

Intelligrator turns MFT interface delivery from a weeks-long engineering engagement into a governed, minutes-long self-service request. Users describe what they need through a guided wizard; the platform validates connectivity up front, prevents duplicates, enforces naming standards, routes the request through approval, and then provisions the interface on Boomi automatically.

Problem

Every new file-transfer interface today requires manual requirement gathering, manual Boomi development, and manual deployment — slow, inconsistent, error-prone, and invisible to requesters until delivery.

Solution

A wizard-driven request portal + approval queue + provisioning engine. Approved requests become live, scheduled MFT interfaces on Boomi Dev with zero hand-written integration code per interface.

Key differentiators

  • Connectivity and folder existence tested at request time, not at delivery time
  • Duplicate-interface detection before submission
  • System-suggested interface names enforcing conventions
  • Full lifecycle visibility: request → approval → build → deployment → run status

Out of scope (v1)

  • Event-triggered (file-arrival) transfers — v1 is scheduled only
  • In-flight transformation/mapping (pass-through file movement only)
  • Auto-promotion beyond Dev (Test/Prod promotion stays governed)

Post-submission lifecycle

  1. Queue: request lands in the admin approval queue (FIFO with priority override), visible to the requester as SUBMITTED.
  2. Review: admin sees the full request, the connection-test evidence, the duplicate-check result, and the generated name. Actions: Approve, Reject (with reason), Return for changes.
  3. Provision: on approval, the provisioning engine configures the interface on Boomi (see §7), applies credentials to the environment via secure extension injection, and attaches the schedule.
  4. Deploy & verify: deployment to the Dev runtime, followed by an automated smoke test (connect to both endpoints, list folders). Status moves to DEPLOYED_DEV.
  5. Notify: requester and point of contact receive confirmation with the interface name, schedule, and a link to the run-status dashboard. Failures at any stage move the request to PROVISION_FAILED with diagnostics for the platform team.

5. User stories

Epic E1 — Access & identity

US-101MUST

As a requester I want to sign in with my corporate identity so that my requests are attributable to me without a separate account.

AC: Given a valid corporate identity, when I access Intelligrator, then I am authenticated via SSO and my display name and email are pre-populated on requests. Sessions expire per security policy.

US-102MUST

As a platform admin I want role-based permissions so that only authorised approvers can action the queue.

AC: Given a user without the Approver role, when they open the approval queue, then access is denied and the attempt is audited. An approver can never approve a request they raised.

Epic E2 — Request wizard

US-201MUST

As a requester I want to capture purpose, business domain, business value and point of contact so that approvers have the business context to make a decision.

AC: Given the requirement step, when any mandatory field is empty, then I cannot advance and the missing field is highlighted. Business domain is selected from a governed list. Point of contact is validated against the directory.

US-202MUST

As a requester I want to enter source server details and test the connection so that I know the interface will work before I submit.

AC: Given valid source details, when I click Test Source Connection, then the system authenticates, verifies the selected folder exists, and displays pass/fail with detail within 15 seconds. The test result (timestamp + outcome) is stored with the request. Given a failed test, then I cannot proceed without a documented override.

US-203MUST

As a requester I want to enter destination details and test the connection so that delivery-side failures are caught at request time.

AC: As US-202 for the destination, plus a write-permission probe (create/delete a zero-byte marker file where policy permits).

US-204SHOULD

As a requester I want to browse and select the source/destination folder so that I don't mistype paths.

AC: Given a successful authentication and an endpoint that permits listing, when I open the folder picker, then I can navigate directories and select one; the selected path populates the form. If listing is not permitted, free-text entry with validation applies.

US-205MUST

As Intelligrator I want to check that the requested interface is new and not a duplicate so that the estate stays clean and costs aren't duplicated.

AC: Given completed source and destination details, when I advance, then a fingerprint of (source host+folder, destination host+folder, file pattern) is checked against the interface registry. On a match, the existing interface name and owner are shown and submission of an identical request is blocked; a "similar" match (same endpoints, different pattern) shows a warning requiring confirmation.

US-206MUST

As a requester I want Intelligrator to suggest the MFT interface name so that naming is consistent without me knowing the standard.

AC: Given a non-duplicate request, then a name of the form MFT-<DOMAIN>-<SOURCE>-<TARGET>-<SEQ> is generated, guaranteed unique, and shown read-only to the requester (approver may override).

US-207MUST

As a requester I want to define file details, a schedule, and error-notification preferences so that the interface runs when and how the business needs.

AC: Given the trigger step, when I pick a frequency, then the equivalent cron and a plain-English description are displayed. Invalid cron is rejected. Notification is explicit opt-in/opt-out with at least one recipient required when opted in.

US-208MUST

As a requester I want a summary of all captured steps before submitting so that I can verify and correct mistakes.

AC: Given the summary step, then every captured value is shown grouped by wizard step with an edit link per group; when I submit, the request becomes read-only, receives a unique request ID, and appears in my tracker as SUBMITTED.

US-209SHOULD

As a requester I want to save a draft and resume later so that I don't lose work while gathering connection details.

AC: Wizard state persists on every step change; drafts appear in "My requests" as DRAFT and reopen at the last completed step.

Epic E4 — Approval workflow

US-401MUST

As an MFT admin I want a queue of submitted requests so that I can process approvals in order and within SLA.

AC: Given submitted requests, then the queue lists them oldest-first with request ID, name, domain, requester, age, and SLA indicator; filters by domain/status/requester are available.

US-402MUST

As an MFT admin I want the full request with connection-test evidence and duplicate-check outcome in one view so that I can decide without leaving the screen.

AC: Given a queue item, when opened, then all wizard data, both test results (with timestamps), and the fingerprint check outcome are displayed; actions Approve, Reject, Return for changes each require/permit a comment and are audited with actor and timestamp.

US-403MUST

As a requester I want to be notified of the approval decision so that I know whether to expect deployment or to amend my request.

AC: On any decision, the requester and point of contact receive a notification containing the decision, comment, and next steps; returned requests reopen as editable drafts.

Epic E5 — Automated provisioning

US-501MUST

As the provisioning engine I want to configure an approved request onto the Boomi MFT engine automatically so that no per-interface development is needed.

AC: Given an APPROVED request, then within 5 minutes the interface configuration is registered, credentials are injected into the secure store (never the application DB), the schedule is attached, and the interface is active on Dev. All provisioning steps are idempotent — re-running for the same request ID never creates duplicates.

US-502SHOULD

As the provisioning engine I want to run a post-deployment smoke test so that the interface is verified working before the requester is told it's live.

AC: After deployment, both endpoints are re-tested from the Dev runtime; success moves the request to DEPLOYED_DEV; failure moves it to PROVISION_FAILED with diagnostics and alerts the platform team, and compensating actions de-schedule the partial deployment.

US-503MUST

As a platform engineer I want provisioning failures surfaced with full diagnostics so that I can remediate quickly.

AC: Failed provisioning records the failing step, error payload, and correlation ID; retry from the failed step is possible without re-approval.

Epic E6 — Observability

US-601MUST

As a requester I want to track my request through every state so that I never need to email anyone for status.

AC: The tracker shows a timeline of state transitions with timestamps and actors, matching the state machine in §9.

US-602MUST

As an interface owner I want to see run history — overall run status and the files transferred successfully so that I can confirm business data is flowing.

AC: Per interface, each execution shows start/end, overall status, file count, per-file name/size/outcome, and error detail for failures. Data retained ≥ 90 days.

US-603MUST

As an auditor I want an immutable audit trail so that every request, decision, and deployment is evidenced.

AC: All lifecycle events are append-only with actor, timestamp, and before/after state; export to CSV is available to the Auditor role.

Three planes: a portal plane (Intelligrator web app + API), a control plane (approval workflow, provisioning engine, interface registry), and an execution plane (the generic MFT engine running on Boomi).

7. Automated Boomi build — the central design decision

Recommendation: configuration-driven provisioning (Option A). Do not generate a new Boomi process per approved request. Deploy one generic MFT engine process once, and make each approved request a row of configuration plus a schedule. "Boomi automatically builds the interface" then means "the engine gains a new configuration entry" — instant, testable, reversible, and free of component sprawl.

Option A — Template process + configuration (recommended)

  1. A single MFT Engine process is built once by the platform team: read interface config → connect source → list/match files → transfer → post-action (archive/delete/rename) → write run records → notify on error (respecting opt-in).
  2. All variability is externalised: endpoints, folders, file patterns, post-actions and notification preferences live in the interface config table; credentials live in environment extensions / vault, resolved at runtime via Dynamic Process Properties.
  3. On approval, the provisioning engine: (1) inserts the config row, (2) writes credentials to the vault/extensions via the Boomi Platform API (EnvironmentExtensions object), (3) creates the per-interface schedule, (4) runs the smoke test, (5) marks the interface active.
  4. Run visibility comes from run records written by the engine itself (interface ID, file names, sizes, outcomes) — richer and more queryable than raw platform execution logs.

8. Design patterns & recommendations

1. Configuration-Driven Integration (metadata-driven engine)

Behaviour is data, not code. Each interface is a validated configuration record interpreted by a generic engine. This is what makes "auto-build" safe and instant, and keeps the Boomi estate flat.

2. Queue-Based Load Levelling + Command

Submission publishes a ProvisionInterface command to the lifecycle topic. The provisioning engine consumes at its own pace; spikes in approvals never overload Boomi Platform API rate limits. Failed commands go to a DLQ for replay.

3. Saga (orchestrated) provisioning with compensation

Provisioning is multi-step (config → secrets → schedule → smoke test). Model it as a saga: each step records completion; on failure, compensating steps (remove schedule, revoke secrets, deactivate config) return the system to a clean state before retry.

4. Idempotent provisioning

The request ID is the idempotency key on every provisioning step. Replays and retries can never create duplicate schedules, config rows, or secrets entries.

5. State Machine for request lifecycle

Explicit states with guarded transitions (see §9). Illegal transitions are impossible by construction; the audit trail is the event log of transitions (event-sourcing-lite).

6. Fingerprint-based duplicate detection

Canonical fingerprint = normalised hash of (source host + folder, destination host + folder, file pattern). Exact match blocks; endpoint-only match warns. Registry is the single source of truth for interface existence.

7. Naming Convention Service

Names are minted centrally: MFT-<DOMAIN>-<SRC>-<TGT>-<NNN>, with a sequence per domain pair. Uniqueness guaranteed by the registry, never by human vigilance.

8. Credential Vaulting / Valet-Key

The portal accepts credentials transiently over TLS, writes them straight to the vault, and stores only a reference. The engine resolves references at runtime. Rotation touches the vault, not the interface.

9. Circuit Breaker + timeout on connection tests

Endpoint tests are bounded (15 s) with a circuit breaker per endpoint host so a dead server can't stall the wizard or exhaust test-service threads.

10. Strangler-fig alignment

Intelligrator provisions new MFT interfaces on Boomi only. Existing legacy transfers remain where they are until migrated; migrated interfaces are re-created as Intelligrator requests, giving them the same governance as net-new ones.

11. Anti-pattern: per-request process generation by default

Component-per-interface (Option B) as the default creates unmaintainable sprawl. Reserve programmatic component generation for exceptional bespoke cases.

12. Anti-pattern: node-local caching of interface config

Boomi's Cache shape is node-local; on clustered Atoms different nodes would see different config. Use the DB-backed config table (with event-driven refresh) as the runtime source.

11. Security & credential handling

  • Credentials never rest in the application database. Captured over TLS → written directly to the vault → only an opaque credentialRef stored. UI never redisplays secrets.
  • Prefer key-based SFTP auth over passwords; support vault references to pre-registered credentials so requesters may not need to know secrets at all.
  • Least privilege: the Boomi Platform API token used by the provisioning engine is scoped to the Dev environment and only the objects listed in §7.
  • Four-eyes: requester ≠ approver enforced at API level, not just UI.
  • Transport: SFTP/FTPS only; plain FTP is not selectable.
  • Audit: append-only event log; connection-test evidence retained with the request for approval accountability.
  • Write probes on destinations are policy-gated (some endpoints prohibit marker files) — configurable per connection profile.
