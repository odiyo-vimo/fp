P-101
Synchronous API

Simple API – Synchronous REST API Proxy

A thin, pass-through REST façade that exposes a backend service through the central iPaaS without altering the payload.

The proxy publishes a managed REST endpoint and forwards the request, largely unchanged, to a single backend system of record (for example an HMRC HoD service). It exists to centralise security, throttling, logging and observability rather than to transform data. Latency overhead is minimal because no mapping is performed; the Atom simply terminates the inbound connection, applies policy, and relays the call synchronously.

Boomi integration flow

When to use

  • You need a governed, monitored entry point in front of an existing service.
  • The consumer and provider already agree on the contract and schema.
  • Centralised auth (OAuth2 / mTLS), rate-limiting and audit are the main requirements.

When to avoid

  • The payload needs transformation or enrichment (use P-102).
  • Multiple backends must be aggregated (use P-103).
  • The call is long-running and should not block the caller (use P-105).

Boomi components used

  • Web Services Server connector (REST listener) as the Start shape
  • API Management / API Gateway policy (OAuth2, mTLS, rate limit)
  • HTTP Client connector to the backend
  • Set Properties shape for correlation ID propagation
  • Try/Catch shape with a Return Documents (fault) path

Potential / alternative components

  • Boomi API Gateway for edge policy enforcement
  • SOAP/Web Services Client connector if the backend is SOAP
  • Notify shape for lightweight access logging
P-102
Synchronous API

Standard API – Synchronous REST API with Data Transformation

A synchronous REST API that maps and reshapes the request/response between the consumer contract and the backend schema.

This is the workhorse API pattern. The Atom terminates the inbound REST call, maps the request into the backend's native structure, invokes the backend, then maps the response back into the published canonical or consumer-facing contract. Business rules, cleansing and value cross-referencing typically sit inside the request/response maps so the consumer is fully decoupled from backend idiosyncrasies.

Boomi integration flow

When to use

  • Consumer contract and backend schema differ and must be reconciled in flight.
  • Field-level validation, enrichment or code translation is required.
  • You want a stable published contract that survives backend change.

When to avoid

  • No transformation is needed (use the lighter P-101).
  • Response times exceed the consumer's synchronous tolerance (use P-105).

Boomi components used

  • Web Services Server connector (REST listener)
  • Map shape (request) and Map shape (response)
  • Business Rules shape for validation
  • Cleanse shape for inbound data hygiene
  • Cross Reference shape for code/value translation
  • HTTP Client connector to the backend
  • Try/Catch with structured fault mapping

Potential / alternative components

  • Data Process shape (custom scripting) for complex reshaping
  • Document Cache for static reference lookups inside the map
  • Set Properties shape for header / correlation handling
P-103
Synchronous API

Aggregation API – Cross-HoD API Integration

A single API call that fans out to several backend services and composes one consolidated response.

The consumer makes one request; the Atom invokes multiple Heads-of-Duty / backend services – in parallel where possible – then merges, de-duplicates and shapes the results into a single response. Partial-failure policy is explicit: the design decides whether a missing contributor degrades gracefully or fails the whole call. Aggregation removes orchestration burden and chattiness from the consumer.

Boomi integration flow

When to use

  • A view needs data from two or more independent backends in one round trip.
  • You want to hide backend topology and reduce consumer-side orchestration.
  • Parallel calls can meet the synchronous latency budget.

When to avoid

  • Contributors are slow or unreliable enough to blow the latency budget (use P-105).
  • Only one backend is involved (use P-102).

Boomi components used

  • Web Services Server connector (listener)
  • Flow Control shape to parallelise backend calls
  • Multiple HTTP Client connector calls (one per HoD)
  • Document Cache to assemble responses for joining
  • Map shape to compose the aggregated response
  • Decision shape for partial-failure handling

Potential / alternative components

  • Branch shape for independent contributor paths
  • Process Call shape to reuse per-backend subprocesses
  • Business Rules shape for completeness checks
P-104
Synchronous API

Experience API – Entitlement-Based Response

Shape and filter the response according to the caller's identity, role or entitlements.

A channel- or persona-specific API that returns only the data the authenticated caller is entitled to see. The Atom resolves entitlements from the token (claims/scopes) or an entitlement service, then prunes, masks or re-shapes fields accordingly. This keeps authorisation logic out of consumers and out of the systems of record, and supports field-level data minimisation for regulated data.

Boomi integration flow

When to use

  • Different callers must receive different views of the same resource.
  • Field-level masking / data minimisation is a compliance requirement.
  • Entitlements are best enforced once, centrally.

When to avoid

  • All callers receive identical responses (use P-102).
  • Authorisation is fully handled at the gateway with no payload shaping needed.

Boomi components used

  • Web Services Server connector (listener)
  • Set Properties shape to extract token claims / scopes
  • HTTP Client connector to an entitlement / profile service
  • Decision shape to branch on entitlement level
  • Map shape with conditional field masking
  • Business Rules shape for redaction policy

Potential / alternative components

  • Cache shape / Document Cache for entitlement results
  • Cross Reference shape for role-to-field mapping
  • Route shape for persona-specific response templates
P-102b
Protocol Mediation

REST-to-SOAP Façade for Legacy HMRC Services

Expose a modern REST contract while the underlying HMRC service speaks SOAP/XML.

Consumers call a clean REST/JSON endpoint; the iPaaS mediates protocol and format, constructing the SOAP envelope, applying WS-Security or mTLS, invoking the legacy service, then parsing the XML response back into JSON. This shields new consumers from WSDL complexity and lets legacy estates be modernised behind a stable REST façade. (Note: this entry shares the P-102 number in the source catalogue – recommend renumbering.)

Boomi integration flow

When to use

  • A new consumer needs REST but the system of record only offers SOAP.
  • You are incrementally strangling a legacy SOAP estate behind REST contracts.
  • WS-Security / SOAP headers must be handled centrally, not by consumers.

When to avoid

  • The backend already exposes REST (use P-101 / P-102).
  • The direction is reversed – SOAP consumer, REST backend (use P-117).

Boomi components used

  • Web Services Server connector (REST listener)
  • Map shape JSON request → SOAP/XML
  • SOAP / Web Services Client connector (imports WSDL)
  • Map shape SOAP/XML response → JSON
  • Set Properties shape for SOAP action / security headers

Potential / alternative components

  • Data Process shape to build or sign the SOAP envelope manually
  • Cross Reference shape for legacy code translation
  • Try/Catch shape mapping SOAP faults to REST error bodies
P-117
Protocol Mediation

SOAP-to-REST Façade for Legacy HMRC Services (TBD)

Allow legacy SOAP consumers to keep calling a WSDL contract while the backend has moved to REST.

The mirror image of the REST-to-SOAP façade. Existing SOAP clients that cannot be changed continue to call a published WSDL; the iPaaS accepts the SOAP request, maps it to the modern REST/JSON backend, and returns a compliant SOAP response. This protects sunk investment in legacy consumers during backend modernisation. Marked TBD in the source catalogue pending confirmation of in-scope legacy consumers.

Boomi integration flow

When to use

  • Legacy SOAP consumers are frozen but the backend is now REST.
  • You must preserve an existing WSDL contract during migration.

When to avoid

  • Consumers can adopt REST directly.
  • Both ends already share a protocol.

Boomi components used

  • Web Services Server connector configured as SOAP (publishes WSDL)
  • Map shape SOAP request → JSON
  • HTTP Client connector to the REST backend
  • Map shape JSON → SOAP response
  • Business Rules shape to honour legacy validation semantics

Potential / alternative components

  • Data Process shape for namespace / envelope handling
  • Cross Reference shape for legacy enumerations
P-105
Asynchronous

Decoupled – Asynchronous Request with Response Polling

Accept a request, return a ticket immediately, process in the background, and let the caller poll for the result.

For long-running or unreliable backend work, the API returns 202 Accepted with a status/location URL instead of blocking. The work is handed to an internal queue or staged store and processed asynchronously; the caller polls a status endpoint until the result is ready. This protects caller threads, smooths load, and decouples consumer SLAs from backend latency. (Source catalogue: 'Pooling' read as 'Polling'.)

Boomi integration flow

When to use

  • Backend processing exceeds a synchronous response budget.
  • You need to absorb spikes and protect caller connections.
  • The consumer can tolerate eventual results via polling.

When to avoid

  • The caller genuinely needs an immediate, in-line answer (use P-102).
  • Push delivery is preferable to polling (use P-111).

Boomi components used

  • Web Services Server connector returning 202 + status URL
  • Boomi Queue / JMS connector to enqueue the request
  • Database (JDBC) operation to persist the ticket and status
  • Separate listener process for the worker (dequeues and runs)
  • Status REST endpoint (second Web Services Server operation)

Potential / alternative components

  • Boomi Event Streams for the internal handoff
  • Set Properties shape to mint the correlation / ticket ID
  • Notify shape for completion signalling
P-106
Data Integration

Database – Direct Database Interaction

Read from or write to a database directly via the iPaaS, without an intermediate service tier.

The Atom connects straight to a database using JDBC to execute parameterised queries, stored procedures or bulk operations. Used for trusted internal data movement, reference-data loads and ETL-style flows where a service façade adds no value. Connection pooling, statement parameterisation and transaction scope must be managed deliberately to protect the database.

Boomi integration flow

When to use

  • Trusted, internal data access where no API tier is warranted.
  • Bulk reads/writes, reference loads or stored-procedure invocation.
  • Performance-sensitive paths that benefit from set-based SQL.

When to avoid

  • External or untrusted consumers need access (front it with P-107).
  • The database schema should be hidden behind a contract.

Boomi components used

  • Database (JDBC) connector – read and write operations
  • Map shape for source-to-target field mapping
  • Cleanse shape for inbound validation
  • Set Properties shape for dynamic SQL parameters
  • Flow Control shape for batched commits

Potential / alternative components

  • Stored-procedure operation on the Database connector
  • Data Process shape for split/zip of large result sets
  • Try/Catch shape with rollback handling
P-107
Data Integration

Database – Database Transactions Exposed as a REST API

Publish a governed REST API over database transactions so consumers never touch the schema directly.

A REST contract is layered over database operations: the Atom validates the request, executes the query or stored procedure within a controlled transaction, and returns a clean resource representation. This gives the database the same security, throttling and observability as any other API while keeping schema and connection details private. Transaction boundaries and idempotency are explicit design concerns.

Boomi integration flow

When to use

  • External or cross-domain consumers need database-backed data via a contract.
  • You want to hide schema and enforce auth/throttling over data access.
  • Write operations need controlled transaction boundaries.

When to avoid

  • Access is internal and trusted (use P-106).
  • The data really belongs behind an existing service of record.

Boomi components used

  • Web Services Server connector (REST listener)
  • Business Rules shape for request validation
  • Database (JDBC) connector with stored-procedure / parameterised SQL
  • Map shape from result set to REST resource
  • Try/Catch shape with transaction rollback and fault mapping

Potential / alternative components

  • Set Properties shape for idempotency keys
  • Cross Reference shape for status / code translation
  • Document Cache for read-through caching of hot lookups
P-108
Event-Driven

Async – Event-Driven – Publish Pattern

Detect a business event and publish it to a broker/topic for any number of interested subscribers.

The producer-side of pub/sub. The Atom captures a change or business event, maps it to a canonical event envelope, and publishes to a topic on a message broker. Delivery is fire-and-forget from the publisher's view: the broker fans the event out to subscribers, who are anonymous and decoupled from the producer. This is the foundation of loosely-coupled, one-to-many event distribution.

Boomi integration flow

When to use

  • One event is of interest to many, possibly unknown, consumers.
  • Producer and consumers must evolve independently.
  • You want broker-managed durability and fan-out.

When to avoid

  • Exactly one known recipient with caller-owned retry (use P-111 webhook).
  • A synchronous answer is required.

Boomi components used

  • Start shape (connector listener or scheduled poll) to detect the event
  • Map shape to the canonical event envelope
  • Boomi Event Streams / JMS / Kafka connector (publish)
  • Set Properties shape for event metadata (type, key, timestamp)
  • Try/Catch shape for publish-failure handling

Potential / alternative components

  • Boomi Queue connector for internal topics
  • Azure Service Bus / AWS SNS connector for cloud brokers
  • Notify shape for publish auditing
P-109
Event-Driven

Async – Event-Driven – Subscribe Pattern

Subscribe to a broker topic/queue, consume events, and process them into a target system.

The consumer-side of pub/sub. A listener process subscribes to a topic or queue, receives events as they arrive, validates and maps them, then applies the change to a target. Acknowledgement, redelivery and dead-letter handling are first-class: a poison message must not stall the stream. Idempotent processing guards against duplicate delivery inherent in at-least-once brokers.

Boomi integration flow

When to use

  • A system must react to events published elsewhere.
  • You need durable, broker-managed consumption with redelivery.
  • Producers are unknown or numerous.

When to avoid

  • You control both ends and a direct call is simpler.
  • Ordering across partitions is strict and cannot be guaranteed.

Boomi components used

  • Boomi Event Streams / JMS / Kafka connector (listener Start shape)
  • Business Rules shape for event validation
  • Map shape to target structure
  • HTTP Client or Database connector to apply the change
  • Try/Catch shape with dead-letter / redelivery routing

Potential / alternative components

  • Set Properties shape for idempotency / de-duplication keys
  • Document Cache to detect duplicate event IDs
  • Notify shape for consumption metrics
P-110
Event-Driven

Async – Event-Driven – Publish Pattern with Polling

Poll a source for changes on a schedule, then publish detected changes as events.

For sources without native event hooks, the Atom polls on a schedule using a high-water-mark (watermark) to detect new or changed records since the last run, then publishes those changes as canonical events. A concurrency lock prevents overlapping runs from double-publishing, and the watermark is advanced only after a successful publish, giving at-least-once delivery with safe restart semantics.

Boomi integration flow

When to use

  • The source has no webhook/CDC but has a queryable change marker.
  • You want event-driven downstreams over a polling-only source.
  • Safe restart and no-double-publish guarantees matter.

When to avoid

  • The source can push events natively (use P-108).
  • Near-real-time latency below the poll interval is required.

Boomi components used

  • Scheduled Start shape
  • Database connector to acquire a lock row and read the watermark
  • Database connector to select changes since the watermark
  • Map shape to canonical event envelope
  • Event Streams / JMS connector (publish)
  • Database connector to advance the watermark post-publish

Potential / alternative components

  • Find Changes shape for change detection
  • Set Properties shape to hold the watermark value in process scope
  • Try/Catch shape to release the lock on failure
P-111
Event-Driven

Push Notification – Webhook Push Notification

Push an event to a single registered recipient endpoint over HTTP, with caller-owned retry.

Unlike pub/sub, a webhook is a one-to-one, registration-based push: the iPaaS calls a recipient's HTTP endpoint when an event occurs. The publisher owns delivery, including retry with back-off, signing/HMAC for authenticity, and dead-lettering after exhausted attempts. Webhooks suit a known partner who wants to be told rather than poll, but lack the broker-managed fan-out and replay of pub/sub.

Boomi integration flow

When to use

  • A single known recipient wants to be notified immediately.
  • The recipient exposes an HTTP callback you can register.
  • Lightweight push without standing up a broker is sufficient.

When to avoid

  • Many/unknown subscribers need the event (use P-108/P-109).
  • You need durable replay and broker-managed redelivery.

Boomi components used

  • Start shape (listener or poll) for event detection
  • Map shape to build the callback payload
  • Data Process / Set Properties shape for HMAC signing
  • HTTP Client connector (POST to the registered URL)
  • Try/Catch shape with retry/back-off and dead-letter routing

Potential / alternative components

  • Database connector for a delivery-attempt / DLQ store
  • Notify shape for delivery auditing
  • Flow Control shape for controlled retry pacing
P-112
File / Batch

File-Based – Managed File Transfer (MFT)

Securely collect, transform and deliver files between parties with full tracking and guaranteed delivery.

The iPaaS picks files up from a source (SFTP/FTPS/disk/cloud store), optionally validates, transforms or splits them, and delivers to one or more targets with checksums, archiving and non-repudiation logging. MFT covers the operational concerns plain transfer ignores: integrity verification, retry, quarantine of bad files, and an auditable trail of every movement – essential in regulated estates.

Boomi integration flow

When to use

  • Bulk or batch data exchanged as files with external/internal parties.
  • Integrity, archiving and audit of every transfer are mandatory.
  • Format conversion or split/merge is needed in transit.

When to avoid

  • Real-time, record-level integration is required (use APIs/events).
  • Volumes are tiny and an API call would be simpler.

Boomi components used

  • SFTP / FTP / Disk connector (inbound get)
  • Data Process shape (split, decompress, validate checksum)
  • Map shape for format conversion
  • SFTP / Disk connector (outbound put) with archive copy
  • Try/Catch shape with quarantine routing for bad files
  • Notify shape for transfer audit logging

Potential / alternative components

  • Amazon S3 / Azure Blob connector for cloud landing zones
  • PGP encryption/decryption via Data Process shape
  • Database connector for a file-tracking ledger
P-118
File / Batch

Batch Processing – Scheduled

Process large data volumes on a schedule in controlled, restartable batches.

A scheduled process extracts a bounded set of records, processes them in chunks, and loads to a target, with checkpointing so a failure resumes rather than restarts from zero. Batch sizing balances throughput against memory and backend limits; because Boomi bills per execution, batching at high volume is also a cost lever. Run windows, parallelism and back-pressure are tuned to protect downstream systems.

Boomi integration flow

When to use

  • High-volume data movement that need not be real-time.
  • Overnight / windowed processing against constrained backends.
  • Cost and throughput benefit from grouping records per execution.

When to avoid

  • Low-latency, record-level reaction is required (use events/APIs).
  • Volumes are small enough that batching adds needless complexity.

Boomi components used

  • Scheduled Start shape with defined run window
  • Database / Disk / SFTP connector for bounded extract
  • Flow Control shape for batch sizing and parallelism
  • Map shape for transformation
  • Database connector for checkpoint / restart markers
  • Try/Catch shape for partial-failure resume

Potential / alternative components

  • Data Process shape for split/zip of large payloads
  • Process Call shape for reusable per-batch logic
  • Notify shape for run-summary reporting
P-113
Cross-Cutting / Default

DEV/TEST – AB Testing Pattern with Synthetic Data

Route a portion of traffic to an alternate implementation or return synthetic data for safe testing.

A default delivery capability: a process can branch between live and candidate implementations (A/B), or return deterministic synthetic data when a test flag or non-production environment is detected. This enables canary comparison, contract testing and load testing without touching real systems of record, and lets consumers develop against a stable mock before the live backend is ready.

Boomi integration flow

When to use

  • You want canary / A-B comparison between implementations.
  • Consumers need a stable synthetic backend during build.
  • Non-production traffic must be isolated from real SoRs.

When to avoid

  • Production traffic must always hit the live backend deterministically.

Boomi components used

  • Web Services Server connector (listener)
  • Decision / Route shape on environment or test header
  • Message shape returning canned synthetic payloads
  • HTTP Client connector for the live path
  • Set Properties shape for traffic-split weighting

Potential / alternative components

  • Branch shape for parallel A/B comparison logging
  • Process Property to hold the active variant
  • Notify shape for variant-level metrics
P-114
Cross-Cutting / Default

Caching in Central iPaaS

Serve hot reference and lookup data from a cache to cut latency and backend load, with controlled invalidation.

A three-tier strategy: static reference data (long TTL), semi-volatile lookups (medium TTL with event-driven invalidation), and transient per-request caching. Because the Boomi Cache shape is node-local and unsuitable for clustered/HA Atoms, the durable tier uses a DB-backed cache table with TTL plus event-driven invalidation. Cache-stampede protection reuses the polling lock-row technique so only one caller refills a cold entry.

Boomi integration flow

When to use

  • Reference/lookup data is read far more often than it changes.
  • Backend calls are expensive and dominate latency.
  • Predictable invalidation on change is achievable.

When to avoid

  • Data must always be strongly consistent and current.
  • The cardinality is too high to cache usefully.

Boomi components used

  • Document Cache / Cache shape for node-local transient caching
  • Database connector for the DB-backed cache table (clustered/HA)
  • Decision shape for hit/miss branching
  • Lock-row technique (Database connector) for stampede protection
  • Event subscriber (P-109) to invalidate on change

Potential / alternative components

  • Set Properties shape for TTL / freshness stamps
  • Redis / external cache via HTTP Client connector
  • Cross Reference table for small static maps
P-115
Cross-Cutting / Default

Logging (Default Capability)

Emit consistent, structured, correlatable logs from every process for observability and audit.

A baseline capability applied to all patterns: each process emits structured log events (correlation ID, pattern ID, status, timings, redacted payload references) to a central log sink. Sensitive data is masked at source. Correlation IDs flow end-to-end so a single business transaction can be traced across processes, queues and backends – a governance requirement in regulated estates.

Boomi integration flow

When to use

  • Always – it is a default, cross-cutting capability.
  • End-to-end traceability and audit are required.

When to avoid

  • Never wholly omitted; only the verbosity/level is tuned per environment.

Boomi components used

  • Set Properties shape to mint/propagate correlation IDs
  • Notify shape for process-level log events
  • Data Process shape to structure and mask log payloads
  • HTTP Client or Queue connector to the central log platform

Potential / alternative components

  • Boomi process reporting / AtomSphere logs for operational view
  • Database connector for an audit ledger table
  • Splunk / ELK / Azure Monitor via HTTP Client connector
P-116
Cross-Cutting / Default

Transaction Management

Coordinate multi-step work so it completes atomically or compensates cleanly on failure.

Because distributed work across heterogeneous backends rarely supports a single ACID transaction, this pattern favours a saga / compensation approach: each step is paired with a compensating action, and a failure triggers rollback of prior steps to restore consistency. Local DB operations use real transactions; cross-system work uses idempotent steps plus compensation. Try/Catch governs the boundaries and error routing.

Boomi integration flow

When to use

  • A business operation spans multiple systems and must stay consistent.
  • Partial completion would leave data in an invalid state.
  • Compensating actions can be defined per step.

When to avoid

  • All work is a single local DB transaction (use native commit/rollback).
  • Steps are naturally independent with no consistency coupling.

Boomi components used

  • Try/Catch shape to bound each step and catch faults
  • Database connector with native transaction for local work
  • Process Call shape for per-step subprocesses + compensations
  • Decision shape to evaluate step outcomes
  • Set Properties shape to track completed-step state

Potential / alternative components

  • Boomi Queue for a durable saga log / outbox
  • Notify shape for compensation auditing
  • Return Documents shape for structured fault output

Appendix

Appendix – Legacy Integrations in the HMRC Estate

The patterns above are realised on top of, and progressively replace, a body of legacy integration in the wider estate. The coexistence approach is a strangler-fig boundary: WebMethods retains ownership of legacy orchestration, Boomi owns new cloud/SaaS integration, and HTTP/REST secured by mTLS or OAuth2 is the only sanctioned crossing point between the two.

ChRIS
Legacy channel/routing infrastructure; integrations are mediated through the iPaaS rather than point-to-point, and migrated behind stable REST contracts where feasible.
WebMethods coexistence
Legacy orchestration stays in WebMethods; new integration lands in Boomi. The boundary is HTTP/REST only, secured with mTLS or OAuth2, so neither platform reaches into the other's internals.
