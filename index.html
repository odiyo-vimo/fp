How to read this. The catalogue’s Integration Modernisation Guidelines (API-First; requirements-driven patterns; preserve async where it adds value; modernise before recreating; adapters for legacy) are correct as principles. This page does the next step: it turns each principle into concrete architecture and a repeatable pattern-selection framework, and it challenges the two places the guidelines are most often misread — so they drive design decisions rather than being read and shelved. Outputs are the canonical patterns K1–K8 from the strategic companion.
1 The five guidelines — operationalised, and challenged
#Guideline (as written)Architecture that delivers itThe challenge — and the answer
G1API-First — where systems can expose/consume APIs, use synchronous REST as the default (simpler, less middleware, better governance, faster delivery).Contract-first APIs (OpenAPI) through a dedicated gateway with dev portal, versioning, quotas. Boomi behind the gateway for orchestration/transformation; direct API-to-API where no mediation is needed.Challenge: “API-First” ≠ “synchronous by default” — conflating them re-introduces the temporal coupling modernisation is meant to remove. Answer: read API-First as contract-first (REST via OpenAPI or events via AsyncAPI). Keep sync REST the default only for interactive request/response with a tight latency budget; hand off to G3 for async. Add a “does this need the iPaaS?” gate to honour “reduced middleware.”
G2Business requirements drive patterns — choose from availability, response time, reliability, consumer scalability, number of consumers, NFRs; not legacy constraints.A published selection framework (§2 + the decision tree) mapping those six dimensions to a canonical pattern. Requirements captured once per interface, driving the choice deterministically and reviewably.Challenge: “number of consumers” keeps appearing as a lead criterion, but a single consumer can still need async. Answer: make coupling & delivery semantics the primary axis; consumer count is secondary. The framework encodes this so the mistake can’t be made by default.
G3Preserve async where it adds value — differing availability windows, long-running, guaranteed delivery, multiple consumers, loose coupling. Use modern messaging, not recreated WebMethods pub/sub.A modern event backbone (log-based streaming: replay, partitioned ordering, at-least-once) with AsyncAPI contracts, idempotent consumers, DLQ + replay. Event Streams integration-scoped; dedicated broker where many non-Boomi systems consume it.Challenge: “don’t recreate WebMethods pub/sub” is easy to violate via old queue/JMS habits. Answer: adopt streaming semantics deliberately — consumers own idempotency and offset, ordering only within a partition key, replay is first-class — not a like-for-like broker port.
G4Modernise interfaces before recreating — assess whether a legacy interface can become an API or event before migrating; modernise the pattern, don’t just move it.An Interface Modernisation Funnel (§5): API? → event? → otherwise adapter, inside a Strangler Fig migration that retires legacy incrementally behind stable strategic contracts.Challenge: assessment can stall delivery, then a lift-and-shift happens anyway under deadline. Answer: time-box the assessment with a default — if it can’t be an API/event in the box, ship an adapter with a recorded sunset trigger. Migration never blocks; the debt is explicit.
G5Use adapters for legacy — systems that can’t adopt strategic standards run through OOTB/custom adapters that isolate protocols, translate to strategic APIs/events, transitional where possible.An Anti-Corruption Layer: the adapter owns protocol mediation and model translation so legacy shapes never leak into strategic contracts. In Boomi this is the thin HTTP boundary to WebMethods/ChRIS (generic client) plus connector-based adapters.Challenge: “transitional” adapters routinely become permanent. Answer: every adapter carries an owner and a documented sunset trigger; adapter debt is tracked on a register and reviewed each governance cycle.
2 Pattern-selection framework (operationalising G2)

Requirement → pattern mapping (the six G2 dimensions)

Requirement dimensionSignalPattern implication
AvailabilityInitiator and target have differing availability windowsAsync / event for temporal decoupling (K1/K2/K3); never hard-couple a sync call to an intermittently available target
Response timeTight, interactive (caller blocks)Sync API (K5); if aggregation threatens the latency budget, use a read model (K6)
Response timeTolerant / long-running (secs–24hrs)Async Request–Reply (K3): 202 + status resource; webhook or poll
ReliabilityGuaranteed delivery / no loss on failureEvent backbone + Outbox (K4) + DLQ + replay; idempotent consumers
Consumer scalabilityMany or highly variable consumersPub/sub fan-out on the event backbone (K1/K2)
Number of consumersSeveral consumers need the same informationEvent backbone (G3). A single consumer does not force sync — use semantics, not count
Non-functionalSecurity, audit, throughput, latency SLA, traceabilityGateway (auth/throttle/quota) + cross-cutting spine (correlation, tracing, DLQ, idempotency)
Guardrail. If two dimensions point in different directions (e.g. tight response time but an intermittently available target), that conflict is the design decision to record — usually resolved by an async request–reply with a read model, not by forcing a fragile synchronous call.
3 Decision tree (in order)

The editable version is the draw.io download at the top. In sequence:

Step 1 · G4/G5 gate
Can the system expose or consume an API or event today?
No → Adapter / ACL (K-Adapter) — wrap it, expose a strategic API/event, record a sunset trigger, then re-enter the flow.
Step 2
Does the initiator need a response?
No → K1 Async Event Publish (fire-and-forget). Multiple consumers / loose coupling / guaranteed delivery (G3) → event backbone K1/K2.
Step 3
Is the response available within an interactive budget (~≤2s) with the caller blocking?
No → K3 Async Request–Reply — 202 Accepted + status resource; webhook or poll.
Step 4 · yes-branch of Step 3
Does it need data from multiple back-ends?
Yes → K6 Aggregated Read — CQRS read model, not live fan-out.
No → K5 Sync API — the API-First default, behind the gateway.
Cross-cutting overrides
Applied regardless of the branch above.
State change must reliably emit an event → K4 Outbox
Bulk / scheduled / file → K7 Batch · K8 MFT (bridge → sunset)
Applies to every path: idempotency keys · correlation ID + distributed tracing · DLQ + replay · ordering / partition keys

5 Interface Modernisation Funnel (G4) & Adapter / ACL (G5)

The funnel — applied to every legacy interface

  1. Can it be a strategic API? If the system can expose/consume a contract-first REST API → modernise to K5 (or K3 if long-running). Preferred outcome under G1.
  2. If not an API, can it be an event? If it fits G3 (loose coupling, guaranteed delivery, multiple consumers, differing availability) → modernise to K1/K2 on the event backbone.
  3. If neither, adapt it. Wrap the legacy interface in an Adapter / ACL that exposes a strategic API/event outward while speaking the legacy protocol inward. Record a sunset trigger.
Anti-lift-and-shift rule. An interface is only re-created as-is on the new platform when the funnel has explicitly rejected both API and event modernisation and an adapter is not viable — and that rejection is documented against the pattern. The default is to modernise, not to move.

Adapter / Anti-Corruption Layer — design rules

  • Isolate legacy protocols: consumers see only strategic APIs/events, never SOAP/WebMethods/ChRIS specifics.
  • Translate legacy models to strategic contracts inside the adapter; legacy field shapes must not leak into canonical schemas.
  • Keep the crossing thin: one sanctioned boundary (generic HTTP client to WebMethods IS), not a native legacy connector.
  • Treat as transitional: every adapter has an owner, a sunset trigger, and a line on the adapter-debt register.
6 Governance guardrails and next moves
  1. Publish the selection framework (§2 + the decision tree) as the mandated route to choosing a pattern — so G2 is enforced, not aspirational.
  2. Add an “API-First, not sync-first” note to G1: default sync for interactive request/response; explicit hand-off to G3 for async; a “does this need the iPaaS?” gate.
  3. Adopt AsyncAPI alongside OpenAPI so events are first-class “APIs” under the API-First regime.
  4. Stand up the adapter-debt register with sunset triggers; review it each governance cycle.
  5. Formalise K3 (Async Request–Reply: 202 + status resource) and K4 (Outbox) as reference patterns — these are where G3 reliability is won or lost.
Note. The guidelines are sound; the value is in making them operational and testable. Where a specific flow contradicts a guideline, record the exception against the pattern — a documented exception is itself good governance.
